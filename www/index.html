Why EasyEnclave

Confidential computing without the distro baggage. No Go, no systemd, no excuses.

📦

Unified Kernel Image

Kernel + initrd + cmdline packed into one EFI binary. One file, one cryptographic measurement. No bootloader chain, no GRUB config, no shim.

🧹

Two Packages, Period

The entire userland is ca-certificates and busybox-static. No package manager in the image. What isn't there can't be exploited.

🔒

dm-verity Rootfs

The root filesystem is integrity-verified at the block level. Tamper with a single byte and the kernel refuses to read it.

⚙

PID 1 Runtime, in Rust

Boots straight to easyenclave. No systemd, no init scripts, no runlevels — and no Go runtime hiding under the hood. A small Rust binary mounts filesystems, reaps zombies, and runs your workload. Nothing else.

🎯

Profile-Driven Builds

make build TARGET=gcp or TARGET=azure or TARGET=local-tdx-qcow2. Each profile defines its kernel cmdline, module set, root strategy, and output format. Add a target in minutes.

✅

TDX Attestation

Generate cryptographic quotes via configfs-tsm. The hardware proves the exact measured image is running unmodified inside a sealed VM. Nonce or report-data binding.

Why an OS-level age gate is a horrible idea

California is moving to make operating systems verify user age. systemd merged the patch. EasyEnclave didn't — here's why that pattern is broken at every layer.

🖥

The OS doesn't know you

Kernels and init systems run processes, not people. Pushing identity verification down to PID 1 means every Linux machine has to know who you are before it will run your code. That's a layering violation, not a feature. Identity belongs to apps and platforms — not the boot path.

📦

It bloats the trusted computing base

An age check pulls in an ID provider, a TLS stack, document parsing, a credential store, telemetry, and a kill switch — all inside the rootfs, all inside the attestation measurement. The whole point of a sealed enclave is to prove a small thing honest. You cannot do that with a million lines of compliance plumbing on the trusted path.

⚙

Servers don't have ages

A confidential VM in a datacenter has no human at the keyboard. Whose ID does the OS check before booting a TDX workload — the cloud provider's? Yours? A tenant who isn't there yet? The mandate is incoherent the second the machine isn't a laptop.

🔓

Compliance creep is a one-way ratchet

Once the hook exists, every jurisdiction wants its own check — region locks, content filters, approved-use lists, export controls. The infrastructure becomes a policy field. There is no version of this where it stays scoped to one rule in one state.

👁

It's a surveillance pipeline by default

Mandatory ID verification at boot means every machine phones home to an identity provider with a hardware-bound credential. That's not a side effect — it's the architecture. Confidential computing exists to keep your workload private from the host; an OS that requires upstream attestation of who you are inverts the threat model.

🔍

And it doesn't even work

A motivated user reflashes the firmware, boots an older kernel, or runs a foreign image. The check protects nobody actually trying to evade it — and breaks every legitimate operator running headless workloads. For confidential computing, that's the entire user population.

EasyEnclave's answer: there is no login, no session, no telemetry, no upstream identity service. PID 1 mounts the rootfs, runs your workload, and reaps zombies. There is no "user" object to attach an age to. Nothing to gate. Nothing to patch.

From Source to Sealed VM

Build a minimal image. Boot it. Deploy workloads. Prove it's real.

1

Build

make build TARGET=gcp assembles a rootfs with mkosi (2 packages), packs it into a Unified Kernel Image, and outputs a disk image. Everything else is excluded.

2

Boot

UEFI firmware loads the UKI directly — no bootloader. The kernel starts, the initrd mounts the dm-verity rootfs, and easyenclave becomes PID 1.

3

Deploy

Send a JSON command over the unix socket. EasyEnclave downloads a static binary from GitHub Releases and runs it as a native process. No containers, no registries.

4

Attest

Request a TDX attestation quote. The hardware generates a cryptographic proof that the exact measured image is running unmodified inside the sealed VM.

Architecture

From firmware to workload — nothing in between that doesn't need to be there.

UEFI firmware
 |
 +-- easyenclave.efi (UKI: kernel + initrd + cmdline)
      |
      +-- easyenclave (PID 1)
      |     |
      |     +-- unix socket: /var/lib/easyenclave/agent.sock
      |     |
      |     +-- workload: your-app (static binary, native process)
      |     |     fetched from GitHub Releases or run as bare command
      |     |
      |     +-- workload: your-client (deploy, attest, logs via socket)
      |
      +-- attestation: configfs-tsm (/sys/kernel/config/tsm/report)

Socket API

Newline-delimited JSON over a unix socket. No HTTP server in the enclave.

health
→ {"method":"health"}
← {"ok":true,"attestation_type":"tdx","workloads":2,"uptime_secs":3600}
deploy
→ {"method":"deploy","github_release":{"repo":"owner/repo",
    "asset":"app"},"cmd":["app"],"app_name":"myapp"}
← {"ok":true,"id":"abc123","status":"deploying"}
attest
→ {"method":"attest","report_data_b64":"AQAB..."}
← {"ok":true,"quote_b64":"AQAAAA..."}

Get Started

Clone, build an image, boot it.

terminal
git clone https://github.com/easyenclave/easyenclave
cd easyenclave

# Build the runtime
cargo build --release

# Build a sealed VM image (GCP, Azure, or local dev)
cd image
make build TARGET=gcp                # GCP GPT disk image
make build TARGET=azure              # Azure fixed-size VHD
make build TARGET=local-tdx-qcow2    # libvirt backing-file qcow2
